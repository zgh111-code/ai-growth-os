## 五、数据库设计（7 张表）

```
user              → 用户（id, username, password, nickname, avatar...）
chat_session      → 聊天会话（id, user_id, title）
chat_message      → 聊天消息（id, session_id, role, content）
daily_review      → 每日复盘（id, user_id, content, mood, AI分析JSON字段）
learning_record   → 学习打卡（id, user_id, category, duration, 联合唯一约束）
expression_record → 表达训练（id, user_id, topic, content, AI评分JSON字段）
project_record    → 项目推进（id, user_id, project_name, status, AI分析JSON字段）
knowledge_chunk   → 知识库分块（id, user_id, filename, content, 全文索引）
```

设计亮点：
- AI 返回的复杂结构（数组形式的建议/任务列表）用 JSON 类型存储，避免了过多的关联表
- 学习打卡表用了 `UNIQUE(user_id, date, category)` 保证每天每类只能打卡一次——**把业务规则写在了数据库层面**

## 六、安全性分析（做得好的地方）

| 安全措施 | 实现方式 |
|----------|----------|
| 密码加密 | BCrypt 不可逆哈希 |
| 无状态认证 | JWT 令牌，7 天过期，拦截器统一校验 |
| 防止越权 | 会话/数据归属校验（只能操作自己的数据） |
| 接口限流 | 聊天 10次/分/用户 + 20次/分/IP；登录 5次/分/IP |
| 密码防泄露 | @JsonIgnore 注解，序列化时跳过 password 字段 |
| 参数校验 | @Valid + @NotBlank，防止空值/恶意输入 |
| 统一错误格式 | 异常不直接暴露给前端，统一包装为 Result |

## 七、架构评价

### 做得好的地方

1. **三层架构清晰**：Controller → Service → Mapper，每层职责明确，这是企业级项目的标准分层方式
2. **模块化增量开发**：4 个成长模块完全独立，增删改不影响原有聊天系统，符合"开闭原则"
3. **配置外置与环境分离**：dev/prod 两套配置，数据库密码不硬编码
4. **统一响应格式**：`Result<T>` 封装 code/msg/data，前端无需关心后端具体异常
5. **文档非常完备**：docs/ 目录下有教程、检查清单、学习笔记，对团队协作和学习者都很友好
6. **SSE 流式实现专业**：用 fetch + ReadableStream 而非 EventSource，因为前者可以携带 Authorization 请求头

### 存在的问题和改进方向

| 问题 | 严重程度 | 建议 |
|------|----------|------|
| 4 个成长模块 CSS 风格与聊天页不一致（用了两套 CSS 变量体系） | 中 | 统一为 Tailwind CSS |
| Lombok 在 Java 25 不兼容，但 pom.xml 仍引入 | 低 | 移除 Lombok 依赖，保持手动 getter/setter |
| RAG 检索用 LIKE 关键词匹配，没有向量语义搜索 | 低（MVP 够用） | 后续可引入 Embedding |
| 聊天历史查询在 Java 内存中分组排序，数据量大时慢 | 中 | 改用 SQL GROUP BY 或加缓存 |
| Dashboard.vue 用原生 fetch 而非统一的 Axios 实例 | 低 | 统一使用封装的 Axios |
| 单元测试只覆盖了 4 个类 | 中 | 补充 Controller 和成长模块的测试 |
| 没有数据库迁移工具（Flyway/Liquibase） | 低（单体项目可接受） | 团队协作时再引入 |

## 十、总结

这是一个**结构清晰、功能完整、安全性到位**的全栈项目。从技术选型（Spring Boot + Vue 3 + MySQL）到架构设计（三层架构 + 模块化增量开发），再到工程规范（多环境配置、部署方案、文档），都体现了良好的工程意识。

对于一个大二学生来说，这个项目已经远超课堂作业水平，达到了一个**合格的个人作品集项目**标准。简历上放这个项目是加分的。

当前阶段的重点不是增加更多功能，而是：
1. **巩固现有知识**——确保每行代码你都理解为什么这样写
2. **补齐短板**——测试、CI/CD、Docker 这些企业必需的技能
3. **学习原理**——不要停留在"会用"，要理解 Spring Boot 自动配置原理、Vue 响应式原理、JWT 工作原理等
