## 变更 1：数据库迁移 — MySQL → PostgreSQL + pgvector

### 背景
原来的 RAG 知识库检索用 MySQL LIKE 做关键词匹配，检索精度差。pgvector 扩展支持原生 `VECTOR(1536)` 类型和余弦相似度运算，是语义搜索的标准方案。

### 具体变更

**依赖层 ([pom.xml](backend/pom.xml))**
- 新增 `org.postgresql:42.7.5` — PostgreSQL JDBC 驱动
- 新增 `com.pgvector:0.1.6` — Java float[] ↔ PG VECTOR 类型转换
- MySQL 驱动保留备用

**配置层**
- [application-dev.yml](backend/src/main/resources/application-dev.yml)：数据源从 `jdbc:mysql://localhost:3306` 改为 `jdbc:postgresql://localhost:5432`，新增 embedding API 配置
- [application-prod.yml](backend/src/main/resources/application-prod.yml)：环境变量前缀从 `MYSQL_*` 改为 `PG_*`，新增 embedding API 配置
- [application.yml](backend/src/main/resources/application.yml)：新增 `type-handlers-package` 用于自动注册 VectorTypeHandler

**数据库层**
- [init_pg.sql](backend/sql/init_pg.sql) **（新建）** — PostgreSQL 完整建表脚本，含 8 张业务表；knowledge_chunk 表使用 `VECTOR(1536)` 类型；注释中预留了 ivfflat 向量索引的创建示例
- [docker-compose.yml](docker-compose.yml) **（新建）** — `pgvector/pgvector:pg16` 镜像，自动挂载 init_pg.sql

## 变更 2：知识库混合检索 — Embedding + 语义搜索

### 背景
旧版 `KnowledgeService.search()` 用字符串 LIKE 做关键词匹配，存在两个问题：
1. 无法理解语义 — "如何提升代码质量" 和 "写出更好的代码" 在关键词层面几乎不匹配
2. `countMatches()` 没做大小写归一化

### 新增文件

| 文件 | 职责 |
|------|------|
| [EmbeddingService.java](backend/src/main/java/com/al/aichat/service/EmbeddingService.java) | 调用 DeepSeek `/v1/embeddings` 获取 1536 维向量，失败自动重试 2 次 |
| [EmbeddingRequest.java](backend/src/main/java/com/al/aichat/dto/EmbeddingRequest.java) | Embedding API 请求体 `{model, input}` |
| [EmbeddingResponse.java](backend/src/main/java/com/al/aichat/dto/EmbeddingResponse.java) | Embedding API 响应体，含嵌套的 data/embedding/usage 结构 |
| [VectorTypeHandler.java](backend/src/main/java/com/al/aichat/config/VectorTypeHandler.java) | MyBatis `BaseTypeHandler<float[]>` — 让 MyBatis-Plus 在读写 knowledge_chunk 时自动完成 float[] ↔ PGVECTOR 转换 |

### 修改文件

**[DeepSeekConfig.java](backend/src/main/java/com/al/aichat/config/DeepSeekConfig.java)**
- 新增 2 个字段：`embeddingApiUrl`、`embeddingModel`

**[KnowledgeChunk.java](backend/src/main/java/com/al/aichat/entity/KnowledgeChunk.java)**
- 新增 `float[] embedding` — 向量数据，`@JsonIgnore` 不返回给前端
- 新增 `Double similarity` — 检索相似度分数，`@TableField(exist = false)` 不入库

**[KnowledgeChunkMapper.java](backend/src/main/java/com/al/aichat/mapper/KnowledgeChunkMapper.java)**
- 新增 `searchByEmbedding()` — pgvector 余弦相似度检索：
  ```sql
  SELECT *, (1 - (embedding <=> #{embedding}::vector)) AS similarity
  FROM knowledge_chunk
  WHERE user_id = #{userId} AND embedding IS NOT NULL
  ORDER BY embedding <=> #{embedding}::vector
  LIMIT #{topK}
  ```

**[KnowledgeService.java](backend/src/main/java/com/al/aichat/service/KnowledgeService.java) — 核心重构**

上传流程增强：
- 分块保存后自动调用 `EmbeddingService.embed()` 生成向量
- embedding 失败不阻塞上传（warn 日志 + 该 chunk embedding 为 null）
- 分块重叠：`OVERLAP = 100`，相邻 chunk 共享 100 字符，减少上下文断裂

检索算法重写 — 从纯关键词升级为混合检索：
1. 查询 → embedding → pgvector 召回 `topK * 2` 候选
2. 对每个候选计算关键词匹配分数（已修复大小写 bug）：
   - 完全匹配 → 1.0
   - 分词匹配 60% + 双字滑动窗口 40%（中文友好）
3. 加权融合：`finalScore = 0.7 * semanticScore + 0.3 * keywordScore`
4. 按融合分数降序返回 topK

新增私有工具方法：
- `vectorToString()` — float[] → pgvector 字符串格式 `[0.1,0.2,...]`
- `computeKeywordScore()` — 0~1 归一化关键词匹配分数（替换旧 `countMatches`）
