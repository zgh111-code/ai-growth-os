## 如何使用这份清单

1. **顺序执行**：按「开发 → 部署 → 上线」三个阶段依次检查，不要跳阶段
2. **逐项打勾**：每检查完一项，在 `[ ]` 中打个 `[x]`
3. **新手提示**：每项都附带了「为什么要检查这个」的解释，方便理解
4. **遇到问题**：查看文末的「常见错误及解决方法」

## 第一阶段：开发阶段检查

> **时间点**：在本地开发机上，代码写完但还没打包之前  
> **目标**：确保代码本身没有错误，功能正常

### 1.1 前端检查

| # | 检查项 | 操作命令/验证方法 | 为什么要检查 | 完成 |
|---|--------|-------------------|-------------|------|
| 1.1.1 | 前端项目能正常启动 | 在 `frontend` 目录下运行 `npm run dev`，浏览器能打开登录页 | 确保依赖安装完整，没有语法错误 | [ ] |
| 1.1.2 | 控制台无报错 | 打开浏览器 F12 → Console，刷新页面，确认没有红色报错 | 红色报错说明代码有问题，比如组件导入路径不对 | [ ] |
| 1.1.3 | API 地址配置正确 | 打开 `frontend/.env.development`，确认 `VITE_API_BASE_URL=http://localhost:8080/api` | 开发环境 API 地址指向本地后端，写错了前端就调不通接口 | [ ] |
| 1.1.4 | 所有页面能正常路由 | 手动访问：`/login`、`/register`、`/chat`、`/history`，确保能正常切换 | 路由配置或路由守卫有问题会导致页面白屏或跳转异常 | [ ] |
| 1.1.5 | 前端能正常打包 | 在 `frontend` 下运行 `npm run build`，看到 `✓ built in Xs` 表示成功 | 如果 build 失败，部署到服务器上也无法运行 | [ ] |
| 1.1.6 | 打包产物生成 | build 后在 `frontend/dist/` 目录下有 `index.html` 和 `assets/` 文件夹 | 没有 dist 目录说明 build 失败，需要修正错误 | [ ] |

| 1.2.9 | 退出登录功能正常 | 点击退出按钮，确认能清除 token 并跳转到登录页 | 确保退出逻辑不会报错 | [ ] |
| 1.2.10 | 修改资料功能正常 | 修改昵称、头像、密码，确认能保存成功 | 用户信息修改功能需要验证 | [ ] |

| # | 检查项 | 操作命令/验证方法 | 为什么要检查 | 完成 |
|---|--------|-------------------|-------------|------|

| 1.3.4 | 数据库连接配置正确 | 检查 `application-dev.yml` 中的数据库地址、用户名、密码是否能连上 | 连不上数据库后端就启动失败 | [ ] |

### 1.4 安全检查

| # | 检查项 | 操作命令/验证方法 | 为什么要检查 | 完成 |
|---|--------|-------------------|-------------|------|
| 1.4.1 | 密码是否加密存储 | 查看 `user` 表的 `password` 字段，应该是 `$2a$10$...` 开头的 BCrypt 加密串 | 明文存储密码是严重的安全漏洞 | [ ] |
| 1.4.2 | JWT Token 是否有效 | 登录后在浏览器 localStorage 中能看到 `token`，复制到 [jwt.io](https://jwt.io) 能解码 | 如果 token 生成异常，所有需要登录的接口都会 401 | [ ] |
| 1.4.3 | 未登录是否能访问受保护页面 | 清除 localStorage 的 token，直接访问 `/chat`，应被重定向到 `/login` | 路由守卫没配置好，未登录用户也能看到聊天页面 | [ ] |
| 1.4.4 | 未登录是否能调接口 | 清除 token 后，用 Postman 调 `/api/chat/history`，应返回 401 | 后端没做认证拦截，未登录也能获取数据 | [ ] |

## 第二阶段：部署阶段检查

> **时间点**：准备把项目部署到生产服务器时  
> **目标**：确保服务器环境准备就绪，配置文件正确

### 2.1 服务器环境检查

| # | 检查项 | 操作命令/验证方法 | 为什么要检查 | 完成 |
|---|--------|-------------------|-------------|------|
| 2.1.1 | 服务器是否安装了 Java 17 | 运行 `java -version`，确认是 `openjdk version "17"` 或更高 | Java 版本不对 JAR 包启动不了 | [ ] |
| 2.1.2 | 服务器是否安装了 MySQL 8.0+ | 运行 `mysql --version`，确认版本号 | 版本太低可能不支持某些 SQL 语法 | [ ] |
| 2.1.3 | 服务器是否安装了 Nginx | 运行 `nginx -v`，确认版本号 | 没有 Nginx 就无法做反向代理和 HTTPS | [ ] |
| 2.1.4 | 服务器是否安装了 Node.js | 运行 `node --version`，确认版本号 ≥ 18 | 前端 build 需要 Node.js（但服务器上只需要 dist 目录就够了） | [ ] |
| 2.1.5 | 服务器时间是否准确 | 运行 `date`，确认与北京时间一致 | 时间不准会导致 JWT Token 验证失败 | [ ] |
| 2.1.6 | 防火墙端口是否开放 | 确认以下端口已开放：`80`(HTTP)、`443`(HTTPS)、`3306`(MySQL 如需远程连接) | 端口没开放，用户无法访问网站 | [ ] |

### 2.2 后端配置检查

| # | 检查项 | 操作命令/验证方法 | 为什么要检查 | 完成 |
|---|--------|-------------------|-------------|------|
| 2.2.1 | 生产环境配置文件就绪 | 确认 `application-prod.yml` 存在且内容完整 | 没有 prod 配置文件，生产环境就无法正常启动 | [ ] |
| 2.2.2 | 所有密文是否改为环境变量 | 检查 `application-prod.yml`，确认以下项使用 `${...}` 占位符：`${MYSQL_USERNAME}`、`${MYSQL_PASSWORD}`、`${JWT_SECRET}`、`${DEEPSEEK_API_KEY}` | 硬编码密码到配置文件里，任何人都能看到 | [ ] |
| 2.2.3 | MySQL 连接是否配置 SSL | 检查 `application-prod.yml` 中数据库 URL 是否包含 `useSSL=true&requireSSL=true` | 生产环境必须使用 SSL 加密数据库连接 | [ ] |
| 2.2.4 | 连接池参数是否合理 | 检查 `hikari` 配置：`maximum-pool-size`、`connection-timeout` 等 | 连接池太小会导致高并发时请求排队超时 | [ ] |
| 2.2.5 | 日志配置是否完善 | 确认日志输出到文件、有滚动策略（按大小/时间切割）、有最大保留天数 | 没有日志文件，出问题后无法排查 | [ ] |
| 2.2.6 | 跨域配置是否需要修改 | 检查 `CorsConfig.java` 中 `addAllowedOrigin` 是否只允许了 `localhost:5173` | 生产环境的前端域名和开发环境不一样，需要更新 | [ ] |

| # | 检查项 | 操作命令/验证方法 | 为什么要检查 | 完成 |
|---|--------|-------------------|-------------|------|

| 2.3.2 | 前端打包 | 在 `frontend` 目录运行 `npm run build`，确保打包成功 | 打包失败部署上去也是空的 | [ ] |
| 2.3.3 | 打包产物确认 | 确认 `dist/` 目录下有 `index.html` 和 `assets/` 子目录 | 有 dist 目录就说明打包成功了 | [ ] |
| 2.3.4 | dist 目录压缩并上传 | 将 `dist/` 目录打包成 zip，上传到服务器 | Nginx 需要指向前端静态文件目录 | [ ] |

### 2.4 数据库配置检查

| # | 检查项 | 操作命令/验证方法 | 为什么要检查 | 完成 |
|---|--------|-------------------|-------------|------|
| 2.4.1 | 生产数据库已创建 | 登录生产 MySQL，创建 `ai_chat` 数据库：`CREATE DATABASE IF NOT EXISTS ai_chat DEFAULT CHARACTER SET utf8mb4;` | 数据库不存在，后端启动时会连不上 | [ ] |
| 2.4.2 | 表结构已创建 | 执行 `backend/sql/init.sql`，确认三张表创建成功 | 没有表，所有接口都会报「表不存在」 | [ ] |
| 2.4.3 | 数据库用户权限正确 | 确认后端使用的 MySQL 用户有 `ai_chat` 数据库的增删改查权限 | 权限不足，SQL 操作会报错 | [ ] |
| 2.4.4 | 数据库字符集正确 | 确认数据库和表的字符集都是 `utf8mb4` | 字符集不对会乱码，或无法存储 emoji | [ ] |

### 2.5 安全配置检查

| # | 检查项 | 操作命令/验证方法 | 为什么要检查 | 完成 |
|---|--------|-------------------|-------------|------|
| 2.5.1 | 生成强 JWT Secret | 生成一个 64 位随机字符串作为 JWT Secret，不要使用开发环境的简单密钥 | JWT Secret 被破解就可以伪造任意用户的 token | [ ] |
| 2.5.2 | JWT Secret 已设为环境变量 | 在服务器上设置 `JWT_SECRET` 环境变量 | prod 配置从环境变量读取，不设置就启动失败 | [ ] |
| 2.5.3 | DeepSeek API Key 已设为环境变量 | 在服务器上设置 `DEEPSEEK_API_KEY` 环境变量 | 不设置后端启动时会报错 | [ ] |
| 2.5.4 | 数据库密码已设为环境变量 | 在服务器上设置 `MYSQL_USERNAME` 和 `MYSQL_PASSWORD` | 这两个是数据库的登录凭证 | [ ] |
| 2.5.5 | 确认 HTTPS 证书就绪 | 准备好 SSL 证书文件（.crt 和 .key），或者准备使用 Let's Encrypt 免费证书 | 没有 HTTPS，用户数据在网络传输中是明文 | [ ] |
| 2.5.6 | 确认 Nginx 配置就绪 | 编写好 Nginx 配置文件，包括：静态文件服务、API 反向代理、HTTPS 配置、HTTP 重定向到 HTTPS | Nginx 配置错误会导致网站无法访问 | [ ] |

### 2.7 接口专项检查

| # | 检查项 | 检查方法 | 为什么要检查 | 完成 |
|---|--------|---------|-------------|------|
| 2.7.1 | 所有接口路径是否统一前缀 `/api` | 检查 `UserController`、`ChatController`、`SessionController` 的 `@RequestMapping` | 路径不统一，前端调接口会 404 | [ ] |

## 附：常见错误及解决方法

### 错误 1：后端启动报 `Failed to bind to 0.0.0.0/0.0.0.0:8080`

**原因**：端口 8080 已被占用  
**解决方法**：
```bash
# 查看谁占用了 8080 端口
netstat -ano | findstr :8080    # Windows
lsof -i :8080                   # Linux

# 杀掉占用进程或修改 application.yml 中的端口号
```

### 错误 2：后端启动报 `Unknown database 'ai_chat'`

**原因**：MySQL 中没有创建 `ai_chat` 数据库  
**解决方法**：
```sql
CREATE DATABASE IF NOT EXISTS ai_chat DEFAULT CHARACTER SET utf8mb4;
```

### 错误 3：后端启动报 `Access denied for user 'root'@'localhost'`

**原因**：数据库用户名或密码错误  
**解决方法**：检查 `application-prod.yml` 中的环境变量是否设置正确

### 错误 4：前端页面白屏，控制台报 `Failed to load resource: net::ERR_CONNECTION_REFUSED`

**原因**：前端调后端 API，但后端没启动或地址不对  
**解决方法**：检查后端是否启动，检查 `.env.production` 中的 `VITE_API_BASE_URL` 是否正确

### 错误 5：前端页面白屏，控制台报 `Uncaught SyntaxError`

**原因**：前端代码打包出错  
**解决方法**：重新运行 `npm run build`，查看错误信息修复

### 错误 6：登录后 `token` 无效，接口返回 401

**原因**：JWT Secret 不一致（开发环境和生产环境用了不同的 Secret）  
**解决方法**：确保生产环境的 `JWT_SECRET` 环境变量已正确设置，且和后端读取的配置一致

### 错误 7：AI 聊天无回复

**原因**：
1. `DEEPSEEK_API_KEY` 环境变量未设置或已过期
2. DeepSeek API 服务不可用  
**解决方法**：检查 API Key 是否有效，查看后端日志中是否有 DeepSeek API 的错误信息

### 错误 8：Nginx 配置错误导致无法访问

**解决方法**：
```bash
# 测试 Nginx 配置语法
nginx -t

# 查看 Nginx 错误日志
tail -f /var/log/nginx/error.log
```
